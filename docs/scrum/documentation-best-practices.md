## 文档使用原则

### 原则 1: 按需查阅，不要记忆

#### 为什么？

- 📚 文档太多，无法全部记住
- 🔄 内容会变化，记忆可能过时
- ⏱️ 查阅文档比记忆更高效

#### 如何做？

**建立个人文档书签系统**

```
浏览器书签文件夹结构:
HermesFlow 文档/
├── 每日必读/
│   ├── 每日工作参考
│   ├── 快速参考
│   └── 故障排查指南
├── 开发/
│   ├── 编码规范
│   ├── Code Review 检查清单
│   └── 开发指南
└── 按需/
    ├── API 设计
    ├── 数据库设计
    └── 系统架构
```

**使用文档搜索功能**

- 📁 **GitHub 搜索**: 在仓库中搜索关键词
  ```
  repo:HermesFlow "Kafka" path:docs/
  ```

- 🔍 **IDE 全局搜索**: VS Code / IntelliJ 的全局搜索
  ```
  Ctrl+Shift+F (Windows/Linux)
  Cmd+Shift+F (Mac)
  ```

- 📖 **Markdown 目录**: 使用每个文档的目录快速定位

**不要试图记住所有细节**

❌ **错误做法**: "我要记住所有 API 的参数"  
✅ **正确做法**: "我知道 API 文档在哪里，需要时查阅"

---

### 原则 2: 及时更新，保持新鲜

#### 为什么？

- 🔒 过时的文档比没有文档更危险
- 🤝 更新文档是对团队的责任
- 📈 及时更新积少成多，容易维护

#### 如何做？

**发现过时内容立即标记**

在文档中添加注释:
```markdown
<!-- TODO: @username 更新为新版本的 API (2025-01-13) -->
```

或创建 GitHub Issue:
```markdown
**标题**: [文档] API 文档过时
**描述**: `api/api-design.md` 中的 `/api/strategies` 接口已变更
**行动**: 更新为新的请求/响应格式
**优先级**: P1 (本 Sprint 修复)
```

**Sprint 结束前更新相关文档**

在 Sprint Retrospective 前完成:
- [ ] 更新 [`progress.md`](../progress.md) - Sprint 总结
- [ ] 更新 API/数据库文档（如有变更）
- [ ] 补充新的 FAQ / 故障排查步骤
- [ ] 记录新的架构决策 (ADR)

**定期运行文档检查清单**

- **每 Sprint**: 运行 [`documentation-ready-checklist.md`](./documentation-ready-checklist.md)
- **每月**: 运行主检查清单 ([`master-checklist-report`](../master-checklist-report-v3.md))
- **每季度**: 全面文档审查和清理

---

### 原则 3: 分享知识，协作改进

#### 为什么？

- 🌟 好的文档是团队的共同成果
- 💡 每个人都有独特的见解
- 🚀 分享加速团队成长

#### 如何做？

**在团队会议中分享有用文档**

每日站会后:
```
"今天我发现 troubleshooting.md 的 PostgreSQL 连接问题部分特别有用，
推荐大家看看"
```

Sprint Review:
```
"本 Sprint 我们新增了 Rust 性能优化文档，展示给大家看看"
```

**提出文档改进建议**

使用 GitHub Issue / Slack:
```markdown
💡 **文档改进建议**

**文档**: development/rust-developer-guide.md
**问题**: 缺少异步编程的最佳实践
**建议**: 添加 Tokio 使用示例和常见陷阱
**影响**: 新手 Rust 开发者上手更快
**我能做**: 我可以提供初稿
```

**贡献新的最佳实践**

发现好的做法，分享给团队:
- 在 Retrospective 中提出
- 补充到相关文档
- 编写技术博客/Wiki

## 文档维护责任

### 责任分工矩阵

| 文档类型 | 主要维护者 | 更新频率 | 审查者 | 重要性 |
|---------|-----------|---------|--------|--------|
| **PRD** | Product Owner | 每月 / 按需 | 团队 + Stakeholders | 🔴 高 |
| **进度文档** | Scrum Master | 每 Sprint | Product Owner | 🔴 高 |
| **架构文档** | Tech Lead | 按需 (ADR) | 架构师 + 团队 | 🔴 高 |
| **API 设计** | 后端开发者 | 每次 API 变更 | Tech Lead | 🔴 高 |
| **数据库设计** | 后端开发者 / DBA | 每次 schema 变更 | Tech Lead | 🔴 高 |
| **开发指南** | Tech Lead | 每季度 | 团队 | 🟡 中 |
| **编码规范** | Tech Lead | 每半年 | 团队 | 🟡 中 |
| **测试文档** | QA Lead | 每 Sprint | QA 团队 | 🔴 高 |
| **部署文档** | DevOps Lead | 按需 | DevOps 团队 | 🔴 高 |
| **故障排查** | DevOps + 开发者 | 持续 | Tech Lead | 🔴 高 |
| **Scrum 文档** | Scrum Master | 每 Sprint | 团队 | 🟡 中 |
| **FAQ** | 全体团队 | 持续 | Scrum Master | 🟢 低 |

---

### 各角色的文档责任

#### Product Owner

**必须维护**:
- PRD 主文档和模块 PRD
- Product Backlog 细化
- 用户故事和验收标准

**更新时机**:
- 需求变更时
- 市场分析后
- 每月例行审查

---

#### Scrum Master

**必须维护**:
- 项目进度文档
- Sprint 回顾记录
- 文档度量和检查清单

**更新时机**:
- 每个 Sprint 结束
- 每日站会后（任务看板）
- 发现文档问题时

---

#### Tech Lead / 架构师

**必须维护**:
- 系统架构文档
- ADR (架构决策记录)
- 技术债务清单
- 编码规范和开发指南

**更新时机**:
- 做出架构决策后 (ADR)
- 技术栈变更时
- 每季度审查规范

---

#### 开发者

**必须维护**:
- API 设计文档（对应自己开发的 API）
- 数据库设计文档（对应自己的 schema 变更）
- 代码注释和 README

**更新时机**:
- 每次 PR 提交时
- API / schema 变更时
- 发现文档错误时

---

#### QA 工程师

**必须维护**:
- 测试策略和测试计划
- 测试用例文档
- 测试报告
- Bug 记录

**更新时机**:
- 每个 Sprint
- 新功能测试后
- 发现 Bug 时

---

#### DevOps 工程师

**必须维护**:
- 部署文档
- 监控和告警文档
- 故障排查指南
- 变更日志

**更新时机**:
- 每次部署后
- 配置变更时
- 发生故障后（补充排查步骤）

## 文档贡献流程

### 流程图

```
发现问题 → 评估优先级 → 创建 Issue/直接修改 → PR Review → 合并更新 → 通知团队
```

---

### 详细步骤

#### 步骤 1: 发现文档问题

**问题类型**:
- ❌ **错误**: 内容不准确、链接失效
- 📅 **过时**: 内容与当前系统不一致
- 📝 **缺失**: 缺少必要信息、示例
- 🎨 **格式**: 格式混乱、不符合规范
- 🔄 **冗余**: 重复内容

---

#### 步骤 2: 评估优先级

| 优先级 | 描述 | 处理方式 | 示例 |
|--------|------|---------|------|
| **P0 - 紧急** | 阻塞工作，立即影响 | 立即修复 | API 文档错误导致调用失败 |
| **P1 - 高** | 严重影响效率 | 本 Sprint 修复 | 缺少关键配置说明 |
| **P2 - 中** | 有影响但不紧急 | 下 Sprint 修复 | 示例代码不够清晰 |
| **P3 - 低** | 优化改进 | 有时间时修复 | 格式美化、错别字 |

---

#### 步骤 3: 创建 Issue 或直接修改

**P0-P1: 创建 GitHub Issue**

```markdown
**标题**: [文档][P1] API 文档错误 - /api/strategies 接口
**描述**:
- **文档**: `docs/api/api-design.md`
- **问题**: `/api/strategies` 的响应格式已变更，但文档未更新
- **影响**: 前端开发者使用旧格式，导致解析错误
- **建议**: 更新为新的响应格式，添加示例

**行动**: @backend-team 请在本周内更新
**验证**: 与实际 API 响应对比
```

**P2-P3: 直接修改并提交 PR**

对于小的改进，可以直接修改:
1. Fork 仓库 或 创建分支
2. 修改文档
3. 提交 PR，标题格式: `[docs] 修复 XX 文档的 YY 问题`

---

#### 步骤 4: Pull Request Review

**PR 描述模板**:

```markdown
## 文档变更说明

**变更类型**: 修复错误 / 更新内容 / 新增文档

**变更文件**:
- `docs/api/api-design.md`

**变更内容**:
- 更新 `/api/strategies` 接口的响应格式
- 添加请求/响应示例
- 修复链接失效

**验证**:
- [ ] 与实际代码一致
- [ ] 链接有效
- [ ] Markdown 格式正确
- [ ] 遵循命名规范

**截图** (如适用):
```

**审查者检查**:
- [ ] 内容准确性
- [ ] 完整性（是否遗漏重要信息）
- [ ] 格式规范（遵循 [`naming-conventions.md`](../naming-conventions.md)）
- [ ] 链接有效性

---

#### 步骤 5: 合并更新

**合并后行动**:
- 更新文档的版本号和更新日期
- 在 Change Log 中记录变更
- 删除文档中的 `TODO` 注释

---

#### 步骤 6: 通知团队

**通知方式**:

在 Slack 发送:
```
📝 文档更新通知

**文档**: API 设计文档
**变更**: 更新了 /api/strategies 接口说明
**链接**: https://github.com/.../docs/api/api-design.md
**影响**: 前端开发者请查看新的响应格式
```

在每日站会提及:
```
"今天更新了 API 文档，/api/strategies 接口有变更，请大家查看"
```

## 文档写作指南

### Markdown 格式规范

#### 标题层级

```markdown
# H1 - 文档标题（每个文档只有一个）

## H2 - 主要章节

### H3 - 子章节

#### H4 - 小节

##### H5 - 更小的节（尽量少用）
```

**原则**:
- 不要跳级（如 H2 直接到 H4）
- H1 是文档标题，内容从 H2 开始
- H4 以下尽量少用，内容太细可能需要拆分文档

---

#### 列表

**无序列表**:
```markdown
- 项目 1
- 项目 2
  - 子项目 2.1
  - 子项目 2.2
- 项目 3
```

**有序列表**:
```markdown
1. 步骤 1
2. 步骤 2
3. 步骤 3
```

**任务清单**:
```markdown
- [ ] 未完成任务
- [x] 已完成任务
```

---

#### 代码块

**行内代码**:
```markdown
使用 `git commit` 提交代码
```

**代码块**:
````markdown
```bash
git add .
git commit -m "Update documentation"
git push origin main
```
````

**指定语言高亮**:
- `bash` / `sh` - Shell 脚本
- `python` - Python 代码
- `java` - Java 代码
- `rust` - Rust 代码
- `json` - JSON 配置
- `yaml` - YAML 配置
- `sql` - SQL 语句

---

```markdown

|-----|-----|-----|

**对齐方式**:

| 左     | 中   | 右     |
```

## 文档审查标准

### 审查检查清单

#### 内容审查

- [ ] **准确性**: 内容与实际系统一致
- [ ] **完整性**: 包含所有必要信息
- [ ] **清晰性**: 表达清晰，易于理解
- [ ] **时效性**: 信息是最新的
- [ ] **示例**: 包含实际可用的示例

#### 格式审查

- [ ] **命名规范**: 遵循 [`naming-conventions.md`](../naming-conventions.md)
- [ ] **Markdown 语法**: 语法正确，渲染无误
- [ ] **目录**: 有清晰的目录结构
- [ ] **链接**: 所有链接有效
- [ ] **代码高亮**: 代码块指定了语言

#### 元信息审查

- [ ] **版本号**: 有版本号
- [ ] **更新日期**: 有最后更新日期
- [ ] **维护者**: 明确维护者
- [ ] **相关文档**: 链接到相关文档

---

### 审查级别

| 文档类型 | 审查级别 | 审查者 | 审查时机 |
|---------|---------|--------|---------|
| **PRD** | 严格 | Product Owner + Stakeholders | 每次更新 |
| **架构文档** | 严格 | Tech Lead + 架构师 | 每次更新 |
| **ADR** | 严格 | 架构师 + 相关开发者 | 创建时 |
| **API/数据库设计** | 严格 | Tech Lead | 每次变更 |
| **开发指南** | 一般 | Tech Lead | 季度审查 |
| **故障排查** | 一般 | DevOps Lead | 持续更新 |
| **FAQ** | 轻度 | Scrum Master | Sprint 审查 |
