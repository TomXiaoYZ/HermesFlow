# Sprint Planning 清单

> **确保 Sprint Planning 会议高效完成的完整检查清单**

---

## 📅 会前准备（Sprint 开始前 2 天）

### Product Backlog 审查

- [ ] **Product Backlog 已按优先级排序**
  - Top 20 Story 清晰可见
  - 优先级与 Q1 2025 路线图对齐
  
- [ ] **Top 20 Story 已细化并准备就绪**
  - 每个 Story 有清晰的描述
  - 每个 Story 有明确的验收标准
  - Epic 已分解为合理大小的 Story（< 2 天工作量）
  
- [ ] **依赖关系已识别并标记**
  - Story 之间的依赖已记录
  - 外部团队依赖已确认
  - 技术依赖（API、数据库）已明确

### 优先级确认

- [ ] **与 Product Owner 确认本 Sprint 的业务价值重点**
  - Sprint 目标草案已准备
  - 关键交付物已明确
  
- [ ] **客户反馈和市场变化已考虑**
  - 最新的客户反馈已整合
  - 竞品动态已分析

### 技术债务审查

- [ ] **查看 `progress.md#技术债务` 部分**
  - 技术债务列表已更新
  - 技术债务优先级已评估
  
- [ ] **评估技术债务影响**
  - 识别阻碍当前开发的技术债务
  - 识别长期影响的技术债务
  
- [ ] **决定本 Sprint 是否处理技术债务**
  - 建议：每个 Sprint 分配 20% 容量用于技术债务

### 架构准备

- [ ] **准备系统架构图**
  - 打印或投影 [系统架构文档](../architecture/system-architecture.md)
  - 准备相关模块的 C4 Component 图
  
- [ ] **准备 ADR 文档**（如有新的架构决策）
  - 审查现有 ADR（`docs/architecture/decisions/`）
  - 识别本 Sprint 需要的新 ADR
  
- [ ] **准备模块文档索引**
  - 打开 [模块索引](../modules/module-index.md)
  - 准备相关模块的 PRD 链接

## 🎯 Sprint Planning 会议（4 小时）

### Part 1: 确定 Sprint 目标（1 小时）

- [ ] **回顾上个 Sprint**
  - 完成的 Story 数量和 Story Points
  - 未完成的 Story 及原因
  - 速度（Velocity）趋势
  
- [ ] **制定 Sprint 目标**
  - 使用模板：`Sprint X 目标: [一句话描述本 Sprint 的核心价值]`
  - 确保 Sprint 目标 SMART（具体、可衡量、可实现、相关、有时限）
  
- [ ] **定义业务价值**
  - 为什么这个 Sprint 对业务重要？
  - 对用户的价值是什么？
  
- [ ] **明确关键交付物**
  - 列出 3-5 个关键交付物
  - 确保可演示（Demo-able）
  
- [ ] **定义成功标准**
  - 至少 3 个可衡量的标准
  - 例如：功能完成、性能达标、测试通过

### Part 2: 选择 Story（1.5 小时）

#### 容量估算

- [ ] **计算团队容量**
  - 公式：`Sprint 容量 = 团队人数 × 工作天数 × 每日可用时间 × 聚焦因子`
  - HermesFlow 示例：
    ```
    团队人数: 6人（2 Rust, 2 Java, 2 Python）
    工作天数: 10天（2周 Sprint，除去节假日）
    每日可用时间: 6小时（除去会议、邮件等）
    聚焦因子: 0.8（考虑中断和意外）
    
    Sprint 容量 = 6 × 10 × 6 × 0.8 = 288 小时 = 36 人日
    ```
  
- [ ] **考虑团队成员休假**
  - 从总容量中减去休假时间
  - 识别关键人员不在的风险
  
- [ ] **考虑会议和其他固定时间**
  - Daily Standup: 10人 × 0.25h × 10天 = 25小时
  - Sprint Review + Retro: 3.5小时
  - Code Review: ~10% 容量

#### Story 选择

- [ ] **从 Product Backlog 顶部开始选择**
  - 按优先级顺序选择
  - 确保每个 Story 都支持 Sprint 目标
  
- [ ] **平衡技术债务和新功能**
  - 新功能: ~80% 容量
  - 技术债务: ~20% 容量
  
- [ ] **考虑技术栈分布**
  - Rust Story: X 小时（Rust 团队容量）
  - Java Story: Y 小时（Java 团队容量）
  - Python Story: Z 小时（Python 团队容量）
  
- [ ] **留有缓冲**
  - 总估算不超过 90% 团队容量
  - 为意外和风险留出空间

#### Story 细化

对于每个选中的 Story，确保：

- [ ] **描述清晰**
  - 使用 "作为 [角色]，我想要 [功能]，以便 [业务价值]" 格式
  
- [ ] **验收标准明确**
  - 至少 3 个验收标准
  - 可测试、可验证
  
- [ ] **技术任务已识别**
  - 任务粒度 < 8 小时
  - 任务负责人明确
  
- [ ] **依赖关系已标记**
  - 依赖其他 Story
  - 依赖外部团队
  - 依赖基础设施
  
- [ ] **相关文档已链接**
  - PRD 链接
  - 架构文档链接
  - API 设计链接

### Part 3: 任务分解和估算（1.5 小时）

#### 任务分解

- [ ] **使用白板或任务看板工具**
  - GitHub Projects / Jira / Trello
  
- [ ] **团队协作分解任务**
  - 每个 Story 分解为 < 8 小时的任务
  - 识别技术任务、测试任务、文档任务
  
- [ ] **任务示例**（以 "实现 RSI 因子计算" Story 为例）：
  - [ ] 设计 RSI 数据结构（Rust）- 2h
  - [ ] 实现 RSI 计算逻辑（Rust）- 4h
  - [ ] 编写单元测试（覆盖率 ≥ 85%）- 3h
  - [ ] 创建 Python FFI 绑定 - 2h
  - [ ] 编写 Python API 文档 - 1h
  - [ ] 集成测试 - 2h
  - [ ] 性能测试（k6）- 2h

#### 估算

- [ ] **使用 Planning Poker 进行估算**
  - 每个人独立估算
  - 同时亮牌
  - 讨论差异（最高和最低估算者说明理由）
  - 重新估算直到达成共识
  
- [ ] **参考历史数据**
  - 使用 HermesFlow 估算参考表（见 [SM-GUIDE](./sm-guide.md#32-估算技术)）
  
- [ ] **记录估算**
  - 在任务看板中记录每个任务的估算
  - 总计每个 Story 的估算

#### 验证容量

- [ ] **总估算 ≤ 团队容量**
  - 如果超出，移除低优先级 Story
  - 或重新评估任务分解
  
- [ ] **各技术栈负载均衡**
  - Rust 团队：X 小时（≤ Rust 团队容量）
  - Java 团队：Y 小时（≤ Java 团队容量）
  - Python 团队：Z 小时（≤ Python 团队容量）

### Part 4: 识别风险和依赖（30 分钟）

#### 技术风险

- [ ] **新技术/库的学习曲线**
  - 识别团队不熟悉的技术
  - 安排学习时间或配对编程
  
- [ ] **性能瓶颈（需要 POC）**
  - 识别性能关键路径
  - 安排性能测试和优化时间
  
- [ ] **第三方 API 限制**
  - Binance API 限流
  - IBKR API 限制
  - 准备限流应对策略
  
- [ ] **跨语言集成复杂度**
  - Rust FFI 绑定
  - gRPC 接口定义
  - 安排集成测试时间

#### 团队风险

- [ ] **关键人员休假**
  - 识别单点故障
  - 安排知识传递或配对编程
  
- [ ] **技能缺口（需要培训）**
  - 识别技能差距
  - 安排培训或配对编程
  
- [ ] **跨团队依赖**
  - 等待其他团队交付
  - 建立沟通机制和检查点

#### 依赖关系

- [ ] **Story 内部依赖**
  - Story A 依赖 Story B
  - 确保 Story B 优先开始
  
- [ ] **外部团队依赖**
  - 识别需要其他团队交付的内容
  - 建立沟通渠道和截止日期
  
- [ ] **基础设施依赖**
  - Kubernetes 集群是否就绪
  - 测试环境是否可用
  - 联系 DevOps 团队确认

#### 缓解措施

- [ ] **为每个高风险项制定缓解计划**
  - 风险描述
  - 影响评估
  - 缓解措施
  - 负责人
  
- [ ] **识别应急计划**
  - Plan B（如果主计划失败）
  - 可以推迟的低优先级任务

### Part 5: 确定 Definition of Done（30 分钟）

#### 代码层面

- [ ] **代码通过 Code Review（至少 1 人审查）**
- [ ] **符合编码规范**（Rust/Java/Python）
  - 参考 [编码规范](../development/coding-standards.md)
- [ ] **无 Linter 错误**
  - Rust: `cargo clippy` 无警告
  - Java: `mvn checkstyle:check` 通过
  - Python: `pylint` 评分 ≥ 8.0
- [ ] **无 Security Scan 漏洞**（Trivy）

#### 测试层面

- [ ] **单元测试通过**
  - Rust: 覆盖率 ≥ 85%
  - Java: 覆盖率 ≥ 80%
  - Python: 覆盖率 ≥ 75%
  
- [ ] **集成测试通过**（如适用）
  - 跨模块集成测试
  - 第三方 API 集成测试
  
- [ ] **性能测试通过**（如适用）
  - 性能基线达标
  - 无性能回归
  
- [ ] **安全测试通过**（如涉及认证/授权/多租户）
  - SQL 注入防护
  - XSS 防护
  - RBAC 权限测试
  - 多租户隔离测试

#### 文档层面

- [ ] **API 文档更新**（如有新 API）
  - OpenAPI/gRPC 规范更新
  - 示例代码更新
  
- [ ] **README 更新**（如有配置变更）
  - 环境变量说明
  - 依赖说明
  
- [ ] **变更日志更新**
  - CHANGELOG.md 更新

#### 部署层面

- [ ] **CI/CD Pipeline 通过**（GitHub Actions）
  - 所有测试通过
  - 安全扫描通过
  - 代码质量检查通过
  
- [ ] **Docker 镜像构建成功**
  - 推送到 ACR
  - 镜像标签正确
  
- [ ] **Helm Chart 更新**（如有变更）
  - values.yaml 更新
  - ConfigMap/Secret 更新
  
- [ ] **在 Dev 环境验证**
  - 部署成功
  - 健康检查通过
  - 烟雾测试通过

#### 验收层面

- [ ] **所有验收标准满足**
  - 功能验收测试通过
  - 性能验收测试通过
  
- [ ] **Product Owner 验收通过**
  - Demo 成功
  - PO 签字确认

## 📤 Sprint Planning 输出

会议结束时，确保有以下输出：

### 文档输出

- [ ] **Sprint 目标文档**
  - Sprint 目标（一句话）
  - 业务价值说明
  - 关键交付物列表
  - 成功标准列表
  
- [ ] **Sprint Backlog**
  - 选中的 Story 列表（带优先级）
  - 每个 Story 的任务分解
  - 每个任务的估算和负责人
  
- [ ] **风险登记表**
  - 识别的风险列表
  - 每个风险的缓解措施
  - 风险负责人
  
- [ ] **依赖关系图**
  - Story 之间的依赖
  - 外部依赖和检查点
  
- [ ] **Definition of Done 确认**
  - 团队共识的 DoD
  - 具体的检查项

### 工具更新

- [ ] **任务看板已更新**
  - GitHub Projects / Jira / Trello
  - 所有 Story 和任务已添加
  - 优先级和标签已设置
  
- [ ] **Sprint 燃尽图已初始化**
  - 理想线已绘制
  - 总 Story Points 已设置
  
- [ ] **文档已更新**
  - `progress.md` 更新（Sprint 目标、待办事项）
  - Sprint Wiki 页面创建

### 沟通

- [ ] **会议纪要已发送**
  - 发送给所有团队成员
  - 抄送 Product Owner 和利益相关者
  
- [ ] **Sprint 目标已公示**
  - 团队 Slack 频道
  - 团队看板（物理或虚拟）
  
- [ ] **下一步行动已明确**
  - 谁先开始哪个 Story
  - 第一个 Daily Standup 时间

---

## ✅ 最终检查

在结束会议前，快速检查：

- [ ] **Sprint 目标清晰且可衡量**
  - 团队所有人理解 Sprint 目标
  - Sprint 目标与公司目标对齐
  
- [ ] **Sprint Backlog 现实可行**
  - 总估算 ≤ 90% 团队容量
  - 有缓冲应对意外
  
- [ ] **风险已识别并有缓解措施**
  - 高风险项有明确的缓解计划
  - 依赖有明确的检查点
  
- [ ] **团队有信心完成承诺**
  - 团队投票：是否有信心完成本 Sprint？
  - 如果信心不足，重新调整 Sprint Backlog
  
- [ ] **Definition of Done 清晰**
  - 所有人理解并同意 DoD
  - DoD 符合质量标准
