## 站会基本设置

### 时间和地点

- **时间**: 每天上午 **10:00**（固定时间，不变）
- **时长**: **15 分钟**（严格限时）
- **地点**: 团队工作区 / Zoom 会议室（链接: ___）
- **频率**: 每个工作日

### 参与者

**必须参加**:
- 开发团队（所有成员）
- Scrum Master

**可选参加**:
- Product Owner
- 利益相关者（仅听，不发言）

### 站会原则

✅ **DO**:
- 准时开始，准时结束（15分钟）
- 站立进行（保持节奏）
- 聚焦 Sprint 目标
- 团队成员之间对话（不是向 SM 汇报）
- 识别障碍
- 可视化进展（看板、燃尽图）

❌ **DON'T**:
- 不要迟到
- 不要深入讨论技术细节（使用 Parking Lot）
- 不要解决问题（站会后处理）
- 不要让站会变成状态汇报会
- 不要超过 15 分钟

## 会议流程

### 会前准备（9:55-10:00）

#### Scrum Master 准备

```bash
# 5分钟准备清单
- [ ] 打开任务看板（GitHub Projects / Jira）
- [ ] 查看昨天的进展（哪些任务移动了）
- [ ] 检查 Sprint 燃尽图（进度是否正常）
- [ ] 准备计时器（15分钟倒计时）
- [ ] 准备障碍记录表
- [ ] 检查网络连接（远程站会）
```

**快速检查**:
- Sprint 目标: ✅ 在轨 / ⚠️ 风险 / 🚨 偏离
- 完成进度: X / Y Story Points（Z%）
- 阻塞任务: X 个
- 昨日完成: X 个任务
- 今日计划: X 个任务

#### 团队成员准备

```bash
# 每个人提前准备好回答三个问题
- [ ] 昨天我完成了什么？
- [ ] 今天我计划做什么？
- [ ] 我遇到什么障碍？
```

### 站会流程（10:00-10:15）

#### 开场（1 分钟）

**Scrum Master 开场词**:
```
早上好！今天是 Sprint X 的第 Y 天。

快速回顾：
- Sprint 目标: [一句话重述]
- 当前进度: X / Y Story Points (Z%)
- Sprint 状态: ✅ 在轨 / ⚠️ 黄色预警 / 🚨 红色风险

让我们开始今天的站会！请大家按顺序分享。
```

#### 轮流发言（10-12 分钟）

**发言顺序**: 
- 按任务看板顺序（从 "In Progress" 列开始）
- 或按名字字母顺序
- 或随机顺序（使用轮盘工具）

**每人发言时间**: 1-2 分钟

使用 [三个问题模板](#三个问题模板)

#### 可视化进展更新（2 分钟）

**Scrum Master 快速更新**:

```markdown
**Sprint 燃尽图**:
- 理想剩余: X Story Points
- 实际剩余: Y Story Points
- 状态: ✅ 正常 / ⚠️ 落后 / 🚨 严重落后

**Sprint 目标风险评估**:
- 🟢 绿色: 目标可达成
- 🟡 黄色: 有风险，需要关注
- 🔴 红色: 目标不可达成，需要调整

**阻塞任务**:
- X 个任务被阻塞
- 需要立即处理: [列出]

**今日焦点**:
- 关键任务: [TASK-ID]（需要今天完成）
- Code Review: [PR-ID]（等待审查）
```

#### 同步和通知（1 分钟）

**今日关键活动**:
- 📅 Code Review 集中时间: 14:00-15:00
- 🚀 部署窗口: 16:00-17:00
- 🎯 技术讨论会: 15:00（主题: ___）

**重要通知**:
- 📢 明天客户 Demo，请准备演示环境
- 🏖️ 张三明天休假，请注意任务交接
- 🔧 测试环境维护: 13:00-14:00

#### 结束（30 秒）

```
感谢大家！站会结束。

Parking Lot 话题稍后讨论：
- 话题 1: [简述]（参与者: @user1, @user2）
- 话题 2: [简述]（参与者: @user3, @user4）

有紧急障碍的同学，请会后立即找我。
```

### 标准模板

```markdown
**[姓名] - [角色]**

- [具体完成的任务/Story]
- [对 Sprint 目标的贡献]
```

## 障碍记录表

### 障碍分类和处理

| 障碍类型 | 处理方式 | 负责人 | 时间目标 |
|---------|---------|--------|---------|
| **技术问题** | 组织技术讨论 | Tech Lead | 当天 |
| **依赖其他团队** | 联系相关团队 | Scrum Master | 24小时 |
| **工具/环境问题** | 联系 DevOps | DevOps | 4小时 |
| **需求不清晰** | 组织 PO 澄清会 | Scrum Master | 当天 |
| **个人技能缺口** | 配对编程/培训 | Tech Lead | 本 Sprint |
| **外部 API 限制** | 调研替代方案 | Tech Lead | 2天 |

### 障碍跟踪模板

```markdown
## 障碍跟踪表 - Sprint X

| ID | 日期 | 报告人 | 障碍描述 | 类型 | 负责人 | 状态 | 解决日期 |
|----|------|--------|---------|------|--------|------|---------|
| B-01 | 01-13 | 李明 | 等待 Data Service API 文档 | 依赖 | @王芳 | 🟡 进行中 | - |
| B-02 | 01-13 | 张伟 | NumPy 版本冲突 | 技术 | @张伟 | 🟢 已解决 | 01-13 |
| B-03 | 01-13 | 刘洋 | ClickHouse 服务不稳定 | 环境 | @DevOps | 🟡 进行中 | - |
```

**状态说明**:
- 🟢 **已解决**: 障碍已移除
- 🟡 **进行中**: 正在处理
- 🔴 **阻塞**: 严重阻塞，需要紧急处理

### 障碍处理 SLA

**紧急障碍** (🔴 红色):
- 阻塞多人或关键路径
- Scrum Master 立即处理
- 目标: **2小时内**解决或升级

**高优先级障碍** (🟡 黄色):
- 阻塞单人但影响 Sprint 目标
- 当天处理
- 目标: **当天**解决

**一般障碍** (🟢 绿色):
- 不紧急，不阻塞进展
- 本 Sprint 内处理
- 目标: **2-3天**解决

## Parking Lot 管理

### 什么是 Parking Lot？

**Parking Lot（停车场）** 是用来记录站会中提出的、需要深入讨论的话题，避免站会超时。

### 识别 Parking Lot 话题

以下情况应放入 Parking Lot：
- ✅ 技术方案讨论（超过 2 分钟）
- ✅ 需要多人参与的决策
- ✅ 复杂问题的根因分析
- ✅ 架构设计讨论
- ✅ 工具选型讨论

### Parking Lot 模板

```markdown
## Parking Lot - [日期]

### 话题 1: RSI 因子性能优化方案

**提出人**: 李明  
**相关人员**: @李明, @Tech Lead, @架构师  
**优先级**: P1  
**预计时长**: 30分钟  
**建议时间**: 今天 14:00

**问题描述**:
RSI 因子当前性能为 5万行/秒，目标是 10万行/秒。需要讨论优化方案（SIMD、并行计算、算法优化）。

**期望输出**:
- 选定优化方案
- 估算优化工作量
- 制定实施计划

---

### 话题 2: 多租户测试策略讨论

**提出人**: 刘洋  
**相关人员**: @刘洋, @QA Team, @Java Team  
**优先级**: P2  
**预计时长**: 45分钟  
**建议时间**: 明天 10:00

**问题描述**:
需要讨论多租户隔离的完整测试策略，包括 PostgreSQL RLS、Redis Key 隔离、Kafka 分区等。

**期望输出**:
- 完整的测试用例列表
- 测试数据准备方案
- 测试自动化计划
```

### Parking Lot 会议流程

**会后 15 分钟内**:
1. Scrum Master 创建 Parking Lot 会议邀请
2. 邀请相关人员
3. 发送议程和期望输出

**Parking Lot 会议**:
- 准时开始和结束
- 聚焦问题，高效决策
- 记录会议纪要
- 明确行动项和负责人

**会后跟踪**:
- 更新任务看板
- 在下次站会通知决策结果

## 常见问题和解决方案

### Q1: 站会总是超时怎么办？

**原因**:
- 深入讨论技术细节
- 参与人员太多
- 没有提前准备

**解决方案**:
- ✅ 严格使用计时器（15分钟倒计时）
- ✅ 技术讨论立即放入 Parking Lot
- ✅ 提醒团队提前准备三个问题
- ✅ Scrum Master 主动打断过长的发言
- ✅ 限制利益相关者发言

---

### Q2: 有人总是迟到怎么办？

**解决方案**:
- ✅ 准时开始（不等迟到的人）
- ✅ 在 Retrospective 中讨论（团队协议）
- ✅ 使用"迟到罐"（迟到者投币 💰）
- ✅ 调整站会时间（如果大多数人不方便）

---

### Q3: 远程团队站会如何高效进行？

**最佳实践**:
- ✅ 使用视频（打开摄像头）
- ✅ 使用虚拟看板（Miro, GitHub Projects）
- ✅ 使用计时器工具（显示倒计时）
- ✅ 提前在 Slack 发送提醒
- ✅ 录制站会（给不能参加的人）

---

### Q4: 有人在站会上保持沉默怎么办？

**原因**:
- 害羞或不习惯
- 没有准备
- 觉得没有重要的事要说

**解决方案**:
- ✅ 主动询问："@张三，你今天有什么更新？"
- ✅ 使用轮流顺序（每个人都要说）
- ✅ 私下了解原因并提供支持
- ✅ 在 Retrospective 讨论站会氛围

---

### Q5: 站会变成了状态汇报会怎么办？

**症状**:
- 所有人都对着 Scrum Master 说话
- 缺乏团队成员之间的互动
- 感觉像是"向上汇报"

**解决方案**:
- ✅ Scrum Master 站在团队外围（不是中心）
- ✅ 使用看板（团队面向看板，而不是 SM）
- ✅ 鼓励团队成员之间提问和互动
- ✅ Scrum Master 保持沉默（除非必要）

---

### Q6: 站会上发现严重问题怎么办？

**示例**: 
- "我们的生产环境宕机了！"
- "关键功能无法完成，Sprint 目标受威胁！"

**应对**:
- 🚨 **立即响应**: "这是紧急情况，我们现在就处理"
- 🚨 **快速结束站会**: "其他人继续，我和 @相关人员 立即处理"
- 🚨 **组织紧急会议**: 立即拉相关人员讨论
- 🚨 **通知 PO**: 如果影响 Sprint 目标

## 📊 站会效果评估

### 每周回顾（Scrum Master）

```markdown
## 站会效果评估 - Week X

**时间控制**:
- 平均时长: ___分钟
- 超时次数: X / 5天
- 目标: ≤ 15分钟

**参与度**:
- 出勤率: ___% （目标: 100%）
- 迟到次数: X 次
- 所有人发言: ✅ / ❌

**障碍识别**:
- 本周识别障碍: X 个
- 当天解决: X 个（___%)
- 本周解决: X 个（___%)

**Parking Lot**:
- 话题数量: X 个
- 及时讨论: X 个（___%)

**团队反馈**:
- 站会有用吗？（1-5分）: ___
- 改进建议: [收集]
```

### 改进行动

如果站会效果不佳，在 Retrospective 中讨论：
- 站会时间是否合适？
- 站会形式是否需要调整？
- 团队协议是否需要更新？
