## 度量概述

### 为什么要度量文档？

**原因**:
- 📊 **可视化**: 数据让文档问题显而易见
- 🎯 **改进导向**: 有度量才有改进目标
- 💡 **价值证明**: 展示文档对团队的价值
- 🔍 **问题发现**: 及早识别文档问题

---

### 度量原则

#### 原则 1: 简单优先

不要追求完美的度量体系，从简单的开始：
- ✅ 手动收集也可以
- ✅ 估算也可以（不需要精确）
- ✅ 定性反馈也重要

#### 原则 2: 行动导向

度量是为了改进，不是为了评分：
- ✅ 发现问题 → 立即行动
- ✅ 度量改善 → 持续优化
- ❌ 只度量不行动 → 浪费时间

#### 原则 3: 定期审查

- 每 Sprint: 快速检查
- 每月: 详细分析
- 每季度: 全面审查

使用 Git 日志统计文档提交：

# 统计过去 30 天的文档变更

### 2. 过时文档数量

#### 度量方法

**定义"过时"**:
- 🔴 严重过时: 6个月未更新，且内容与实际不符
- 🟡 可能过时: 3-6个月未更新
- 🟢 新鲜: 3个月内更新

**检查方式**:

```bash
# 查找 6 个月未更新的文档
find docs/ -name "*.md" -mtime +180 -ls

# 查找 3-6 个月未更新的文档
find docs/ -name "*.md" -mtime +90 -mtime -180 -ls
```

---

#### 示例数据

**2025年1月 - 过时文档统计**

| 状态 | 数量 | 占比 | 文档示例 |
|------|------|------|---------|
| 🔴 严重过时 | 2 | 4% | `archived/` (已归档) |
| 🟡 可能过时 | 5 | 10% | `deployment/docker-guide.md` |
| 🟢 新鲜 | 43 | 86% | 大部分文档 |
| **总计** | **50** | **100%** | - |

---

#### 分析和行动

**目标**: 🔴 严重过时 < 5%, 🟡 可能过时 < 15%

**行动**:
1. 审查 5 个可能过时的文档，确认是否需要更新
2. 对于长期不变的文档（如稳定的规范），标记为"稳定版本"

## 改进建议

### 基于度量的改进框架

#### 1. 发现问题

**通过度量识别问题**:
- 某文档访问量突然下降 → 可能被更好的文档替代，或内容过时
- 某文档访问量激增 → 可能最近相关工作多，或遇到问题
- 高频搜索无对应文档 → 缺失文档
- 文档错误报告增多 → 质量下降

---

#### 2. 分析根因

**5个为什么分析法**:

示例：
- **问题**: API 文档过时
- **为什么 1**: 开发者修改 API 后未更新文档
- **为什么 2**: 开发者不知道需要更新文档
- **为什么 3**: Definition of Done 中未包含文档更新
- **为什么 4**: 缺少文档更新检查机制
- **为什么 5**: 团队对文档重要性认识不足

**根因**: 缺少机制和意识

---

#### 3. 制定方案

**针对根因的解决方案**:
1. 在 Definition of Done 中加入"文档已更新"
2. 在 Code Review Checklist 中加入"检查文档更新"
3. 在 Sprint Retrospective 中讨论文档质量
4. 奖励优秀文档贡献

---

#### 4. 实施和跟踪

**行动计划**:
- **短期 (本 Sprint)**: 更新 DoD 和 Code Review Checklist
- **中期 (下月)**: 在 Retrospective 中专题讨论文档
- **长期 (本季度)**: 建立文档贡献奖励机制

**度量改善**:
- 目标: 文档错误报告数量下降 50%
- 跟踪: 每月检查进展
- 调整: 如果效果不佳，调整方案

---

### 改进优先级矩阵

| 影响 / 成本 | 低成本 | 中成本 | 高成本 |
|------------|--------|--------|--------|
| **高影响** | 🔴 立即做<br>例: 修复链接失效 | 🟡 计划做<br>例: 创建缺失文档 | 🟢 评估后做<br>例: 视频教程 |
| **中影响** | 🟡 有空做<br>例: 格式美化 | 🟢 排期做<br>例: 文档重构 | ⚪ 暂不做<br>例: 文档平台升级 |
| **低影响** | 🟢 随时做<br>例: 修正错别字 | ⚪ 可不做<br>例: 过度细化 | ⚪ 不做<br>例: 不必要的功能 |

---

### 持续改进循环

```
度量 → 分析 → 改进 → 验证 → 度量 (循环)
  ↓       ↓       ↓       ↓
 发现   理解   行动   确认
 问题   根因   计划   效果
```

**关键**:
- 不要停留在度量，要行动
- 不要一次改太多，小步快跑
- 不要忘记验证效果，及时调整

### 自动化脚本示例

#### 检查过时文档

```bash
#!/bin/bash
# check-outdated-docs.sh

echo "📊 检查过时文档 (>90天未更新)"
echo "================================"

find docs/ -name "*.md" -mtime +90 -exec sh -c '
    file="$1"
    days=$(( ($(date +%s) - $(stat -f %m "$file")) / 86400 ))
    echo "$days 天前: $file"
' _ {} \; | sort -n -r
```

---

#### 检查链接有效性

```bash
#!/bin/bash
# check-links.sh

echo "🔗 检查文档链接有效性"
echo "================================"

npx markdown-link-check docs/**/*.md --config .markdown-link-check.json
```

配置文件 `.markdown-link-check.json`:
```json
{
  "ignorePatterns": [
    {
      "pattern": "^http://localhost"
    }
  ],
  "timeout": "20s",
  "retryOn429": true,
  "retryCount": 3
}
```
