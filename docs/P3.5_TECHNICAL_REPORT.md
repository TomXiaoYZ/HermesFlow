## P0: Z-Score Normalization for LongShort Mode

### Commit: `53338d4`

### Problem Statement

LongShort mode was producing **identical results to LongOnly** — no short positions were ever generated. The root cause was sigmoid compression of raw VM signal outputs.

**The Bug**: The signal normalization pipeline applied `sigmoid(raw_signal)` uniformly to both LongOnly and LongShort modes. For typical formula outputs (which resemble financial returns in the range `[-0.05, 0.05]`), sigmoid maps values to a narrow band around 0.5:

```
sigmoid(0.03)  ≈ 0.5075
sigmoid(-0.03) ≈ 0.4925
sigmoid(0.00)  = 0.5000
```

The adaptive lower threshold (30th percentile of sigmoid values, clamped to `[0.20, 0.48]`) was unreachable because the entire signal distribution compressed into `[0.4925, 0.5075]`. The clamp forced lower_threshold to 0.48, but no sigmoid value could fall below that — **zero short signals were ever generated**.

### Solution: Dual Normalization Path

```
Signal Normalization (per mode):
  ├─ LongOnly:  sigmoid(raw) → threshold via percentile of sigmoid values
  │             Range: (0, 1), upper ∈ [0.52, 0.70]
  │             (unchanged — zero regression risk)
  │
  └─ LongShort: z-score(raw) = (raw - μ) / σ
                Range: (-∞, +∞), centered at 0
                upper ∈ [0.1, 2.0], lower ∈ [-2.0, -0.1]
                Symmetric around zero → natural long/short split
```

### Code Changes (5 functions affected)

All 5 fitness evaluation paths were updated to branch on `StrategyMode`:

| Function | Purpose | Change |
|----------|---------|--------|
| `psr_fitness` | In-sample K-fold PSR | Z-score for LongShort |
| `pnl_fitness` | In-sample PnL fitness | Z-score for LongShort |
| `psr_fitness_oos` | Out-of-sample PSR | Z-score with train-derived (μ, σ) |
| Walk-forward loop | Walk-forward OOS evaluation | Z-score with per-step train (μ, σ) |
| `detailed_backtest_simulation` | Full equity curve sim | Z-score for LongShort |

### New Functions Added

```rust
/// Compute mean and std of raw signal. std floored at 1e-10.
fn zscore_params(sig: &[f64], start: usize, end: usize) -> (f64, f64)

/// 70th percentile of z-scored signal, clamped to [0.1, 2.0].
fn adaptive_threshold_zscore(sig, start, end, mean, std) -> f64

/// 30th percentile of z-scored signal, clamped to [-2.0, -0.1].
fn adaptive_lower_threshold_zscore(sig, start, end, mean, std) -> f64
```

### Look-Ahead Bias Prevention

For walk-forward and OOS evaluation, `(μ, σ)` are computed from the **train window only** and applied to the **test window**:

```rust
// Walk-forward: train-derived z-score params passed to OOS evaluator
let (mean, std) = zscore_params(sig_slice, train_start, train_end);
let (psr, trade_count, active_count) = self.psr_fitness_oos(
    sig_slice, ret_slice, test_start, test_end,
    mode, upper, lower,
    Some((mean, std)),  // ← train-derived, applied to test
);
```

## P2: MTF Data Loading Parallelization

### Commit: `dc30047`

### Problem Statement

`load_data_multi_timeframe` loaded candles sequentially:
```
for symbol in symbols:                  // sequential
    for resolution in resolutions:      // sequential
        fetch_candles(symbol, res)      // await (DB query)
        if Polygon: fetch_spy_ref()     // await (DB query)
    compute_features()                  // CPU
    forward_fill_align()                // CPU
```

For 13 symbols × 3 resolutions = 39 DB queries + up to 39 SPY reference queries = **78 sequential DB round-trips** at startup. With TimescaleDB query latency of ~5-20ms each, this added 0.4-1.6s of sequential I/O before evolution could begin.

### Architecture Change

```
BEFORE (sequential):                 AFTER (parallel):
┌──────────────────┐                ┌──────────────────────────┐
│ for symbol       │                │ stream::iter(symbols)    │
│   for resolution │                │   .map(|sym| async {     │
│     DB query     │──serial──→     │     tokio::spawn(res1)  │──concurrent──→
│     DB query     │                │     tokio::spawn(res2)  │
│   features()     │                │     tokio::spawn(res3)  │
│   align()        │                │     tokio::spawn(spy1)  │
│   cache.insert() │                │     tokio::spawn(spy2)  │
└──────────────────┘                │     tokio::spawn(spy3)  │
                                    │     features()           │
                                    │     align()              │
                                    │   })                     │
                                    │   .buffer_unordered(8)   │
                                    │ cache.insert_all()       │
                                    └──────────────────────────┘
```

**Concurrency model**:
- **Symbol level**: `buffer_unordered(8)` — up to 8 symbols processed concurrently
- **Resolution level**: `tokio::spawn` per resolution per symbol — all 3 resolutions fetched in parallel
- **SPY reference**: Also `tokio::spawn` per resolution (Polygon only)
- **Feature computation**: Sequential within each symbol (CPU-bound, no async benefit)
- **Concurrency limit = 8**: Prevents DB connection pool exhaustion (default pool = 10)

### Key Refactoring: `&self` → Standalone Functions

The original methods `fetch_candles_for_resolution` and `load_reference_close` were `&self` methods on `Backtester`. This created a borrow conflict: `buffer_unordered` requires `'static` futures, but `&self` borrows are non-`'static`.

**Solution**: Extract to standalone `async fn` with explicit parameters:

```rust
// BEFORE (method — can't move into async closure)
async fn fetch_candles_for_resolution(&self, symbol: &str, ...) -> ...

// AFTER (standalone — cloneable params, 'static-compatible)
async fn fetch_candles_parallel(
    pool: &PgPool,       // PgPool is Clone + 'static
    exchange: &str,
    symbol: &str,
    resolution: &str,
    days: i64,
) -> anyhow::Result<Option<CandleArrays>>
```

### New Type Alias

```rust
/// OHLCV arrays + timestamps returned from a resolution fetch.
type CandleArrays = (
    Array2<f64>, // close
    Array2<f64>, // open
    Array2<f64>, // high
    Array2<f64>, // low
    Array2<f64>, // volume
    Array2<f64>, // liquidity
    Array2<f64>, // fdv
    Array2<f64>, // amount
    Vec<i64>,    // timestamps
);
```

This replaces the previous 9-element inline tuple return type, improving readability.

### Parallelism Structure

```rust
let results: Vec<Option<(String, CachedData)>> = stream::iter(symbols.iter().cloned())
    .map(|symbol| {
        let pool = pool.clone();
        // ... clone other params ...
        async move {
            // Phase 1: Parallel DB queries (all resolutions + SPY refs)
            let mut futures = Vec::with_capacity(resolutions.len());
            for resolution in &resolutions {
                futures.push(tokio::spawn(async move {
                    fetch_candles_parallel(&pool, &exchange, &symbol, &resolution, days).await
                }));
            }
            // Await all resolution fetches
            for handle in futures { candle_results.push(handle.await??); }

            // Phase 2: Parallel SPY reference fetches (Polygon only)
            // ... similar tokio::spawn pattern ...

            // Phase 3: Sequential feature computation (CPU-bound)
            for (res_idx, candle_data) in candle_results.into_iter().enumerate() {
                let features = FeatureEngineer::compute_features_from_config(...);
                res_data.push((features, timestamps));
            }

            // Phase 4: Stack & align features
            // ... forward_fill_align, future returns computation ...

            Ok(Some((symbol, cached_data)))
        }
    })
    .buffer_unordered(8)
    .collect::<Vec<_>>()
    .await
    .into_iter()
    .collect::<Result<Vec<_>, _>>()?;
```

### Performance Expectations

| Metric | Before (sequential) | After (parallel) | Speedup |
|--------|---------------------|-------------------|---------|
| DB queries per symbol | 3-6 serial | 3-6 concurrent | ~3-6× per symbol |
| Symbols concurrency | 1 | Up to 8 | ~8× throughput |
| Theoretical max speedup | — | — | ~13× (if DB-bound) |
| Practical estimate | ~1.0-1.6s | ~0.1-0.3s | **3-5×** |

**Why practical < theoretical**:
1. Feature computation is CPU-bound (unchanged)
2. DB connection pool (10) limits true parallelism
3. `buffer_unordered(8)` caps concurrent symbols
4. TimescaleDB may serialize heavy queries on the same hypertable
