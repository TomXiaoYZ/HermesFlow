# P6 Design: Full-Stack Evolution Upgrade + Live Paper Trading

> Status: **DESIGN APPROVED** | Date: 2026-02-23
> Inputs: Original P6 Roadmap + Gemini P5 Architecture Review + P5 Future Work

## 1. Context

P0-P5 established HermesFlow's full strategy evolution pipeline: OOS evaluation (P0), factor enrichment (P1), LLM oracle (P2), multi-timeframe stacking (P3), adaptive thresholds (P4), and HRP portfolio ensemble (P5). The system can evolve, evaluate, and allocate capital to strategies in simulation.

P6 bridges simulation and reality: hardening the evolution engine, validating strategies in live markets via paper trading, and upgrading the search algorithm from heuristic GP to MCTS symbolic regression.

## 3. P6a — Foundation Hardening & Evolution Enhancement

### E1. Atomic Semantic Block Encapsulation

**Problem**: LLM Oracle generates semantically coherent RPN fragments (e.g., Bollinger Band breakout logic). ALPS crossover/mutation operators violently sever these blocks, destroying prior knowledge.

**Design**:
- Add `block_mask: Vec<u8>` to genome representation. Each element indicates block membership (0 = unprotected, N = block ID N).
- LLM Oracle annotates semantic block boundaries when injecting seeds. The genome decoder parses `[BLOCK_START_N ... BLOCK_END_N]` markers from LLM output.
- **Crossover modification**: Cut-point selection skips positions inside blocks. Cuts only at block boundaries or unprotected regions.
- **Mutation modification**: Inside blocks, allow constant perturbation (nudge numerical values ±10%). Forbid structural mutation (operator replacement, token deletion/insertion).
- **Block decay**: If an individual carrying a block shows no fitness improvement for 20 consecutive generations, block protection is lifted. Prevents fossilization of outdated semantic structures.
- **Backward compatibility**: Genomes without `block_mask` (all existing genomes) default to fully unprotected (empty mask).

**Files**: `strategy-generator/src/genetic.rs`, `strategy-generator/src/llm_oracle.rs`, `strategy-generator/src/genome_decoder.rs`, `backtest-engine/src/genome.rs`

**Verification**: Unit tests for crossover respecting block boundaries, mutation constrained inside blocks, block decay after N generations.

### E2. Protected-Operation Penalty Mechanism

**Problem**: Protected division, protected log, etc. can produce low-volatility pseudo-signals that receive deceptively high Sharpe ratios. The VM's safety net becomes a fitness landscape deception source.

**Design**:
- Add `protection_trigger_count: u32` to VM execution context. Increment on every protected-op activation (zero denominator, negative sqrt input, etc.).
- After execution, compute `protection_ratio = protection_trigger_count / total_ops_executed`.
- Fitness penalty: when `protection_ratio > threshold` (default 0.15):
  ```
  adjusted_psr = raw_psr * max(0.1, 1.0 - protection_ratio)
  ```
- Configuration: `config/generator.yaml` → `vm.protection_penalty_threshold: 0.15`
- Logging: Track protection ratio in `strategy_generations.metadata` for monitoring.

**Files**: `backtest-engine/src/vm.rs`, `strategy-generator/src/backtest/psr_fitness.rs`, `config/generator.yaml`

**Verification**: Test that a genome triggering 50% protected ops receives near-zero PSR. Test that a clean genome is unaffected.

### E3. SIMD Vectorization (Conditional)

**Problem**: RPN VM processes 75-dimensional feature vectors element-by-element. SIMD can batch basic arithmetic.

**Design**:
- **Gate**: Profile VM hot path with `cargo flamegraph`. Only proceed if ALU operations dominate (> 40% of execution time).
- If confirmed: implement SIMD paths for Add, Sub, Mul, Div, Abs using `std::arch::aarch64` (NEON for ARM Mac) or `std::arch::x86_64` (AVX2 for Linux deploy).
- Fallback: scalar path retained for non-SIMD platforms via `#[cfg(target_arch)]`.
- This is an optimization, not a blocker for P6a completion.

**Files**: `backtest-engine/src/vm.rs`

### G1. ndarray Zero-Copy Multi-Timeframe Alignment

**Problem**: Multi-timeframe alignment (P3) may perform unnecessary data cloning when mapping 1d data to 1h time axis.

**Design**:
- Audit `strategy-generator/src/backtest/` for `.to_owned()`, `.clone()`, and `Array::from_vec()` in multi-timeframe paths.
- Replace data duplication with `ArrayView` + stride manipulation: use `ndarray::ArrayView::from_shape()` with custom strides to map 1 daily value across 24 hourly slots without memory allocation.
- For broadcasting: use `.broadcast()` instead of manual expansion.
- Benchmark before/after with `criterion` to confirm improvement.

**Files**: `strategy-generator/src/backtest/`, `backtest-engine/src/factors/`

### G2. rayon CPU Isolation

**Problem**: ALPS fitness evaluation (CPU-bound) runs on Tokio async threads, potentially starving DB I/O and Redis pub/sub.

**Design**:
- Add `rayon` dependency to `strategy-generator/Cargo.toml`.
- Wrap `evaluate_generation()` calls in `rayon::spawn()` with `tokio::sync::oneshot` for result notification back to the async context.
- Tokio handles only: DB queries (SQLx), Redis pub/sub, HTTP API, health checks.
- rayon handles: genome fitness evaluation, HRP computation, ensemble rebalance math.
- rayon thread pool size: `RAYON_NUM_THREADS` env var (default: num_cpus - 2, reserving cores for Tokio).

**Files**: `strategy-generator/src/main.rs`, `strategy-generator/Cargo.toml`

### G3. rkyv Zero-Copy Deserialization (Conditional)

**Problem**: Redis high-frequency market data uses `serde_json` deserialization with heap allocations.

**Design**:
- **Gate**: Profile data-engine and strategy-engine Redis consumption paths. Only proceed if serde_json deserialization accounts for > 15% of processing time.
- If confirmed: serialize `StandardMarketData` with rkyv for Redis pub/sub. Consumers read directly from the byte buffer without allocation.
- Version compatibility: include a 1-byte version prefix in messages. Support dual-mode reading (rkyv + JSON fallback) during migration.

**Files**: `data-engine/src/storage/redis.rs`, `strategy-engine/src/`, `common/Cargo.toml`

### F1. EWMA Covariance Matrix

**Problem**: HRP uses equal-weight rolling window for correlation, slow to adapt to regime changes.

**Design**:
- New function `ewma_correlation_matrix(returns: &Array2<f64>, lambda: f64) -> Array2<f64>` in `hrp.rs`.
- Exponentially weighted moving average with decay factor λ (default 0.94, per RiskMetrics standard).
- Weight for observation at time t: `w_t = λ^(T-t) * (1-λ) / (1 - λ^T)`.
- Configuration: `ensemble.covariance_method: "ewma"` (alternatives: "equal_weight").
- `ensemble.ewma_lambda: 0.94`.
- Retain equal-weight method as fallback and for A/B comparison.

**Files**: `strategy-generator/src/backtest/hrp.rs`, `config/generator.yaml`

**Verification**: Test that EWMA with λ=1.0 degenerates to equal-weight. Test that recent observations have higher weight.

### F2. Portfolio Turnover Cost Modeling

**Problem**: Shadow equity does not reflect portfolio-level rebalance costs.

**Design**:
- On each rebalance, compute `turnover = sum(|w_new[i] - w_old[i]|) / 2` (range 0-1).
- Subtract estimated cost: `cost = turnover * cost_rate * portfolio_value`.
- Cost rates by venue: `{ "IBKR": 0.0001, "Binance": 0.001, "Futu": 0.0003, "Solana": 0.001 }`.
- Track cumulative turnover cost in `portfolio_ensembles.metadata`.
- New metric in ensemble API response: `total_turnover_cost`, `avg_turnover_per_rebalance`.

**Files**: `strategy-generator/src/main.rs` (rebalance loop), `strategy-generator/src/backtest/ensemble_weights.rs`

## 4. P6b — Live Paper Trading & Turnover Deadzone

### B1. Strategy Deployment Pipeline

**Problem**: No automated path from strategy-generator (evolved strategies) to strategy-engine (live signal generation).

**Design**:

**New DB table** (`031_deployed_strategies.sql`):
```sql
deployed_strategies (
    id UUID PK DEFAULT gen_random_uuid(),
    exchange TEXT NOT NULL,
    symbol TEXT NOT NULL,
    mode TEXT NOT NULL,          -- 'long_only' | 'long_short'
    genome INT[] NOT NULL,
    threshold_config JSONB,
    ensemble_version INT,
    oos_psr FLOAT8,
    deployed_at TIMESTAMPTZ DEFAULT now(),
    status TEXT DEFAULT 'active', -- active | paused | retired
    UNIQUE(exchange, symbol, mode)
)
```

**Flow**:
1. Ensemble rebalance completes → writes qualifying strategies to `deployed_strategies` (UPSERT on exchange+symbol+mode).
2. strategy-engine polls `deployed_strategies` every 30 min (configurable) for `status = 'active'`.
3. strategy-engine loads genome + threshold config, initializes VM instance per strategy.
4. VM processes real-time market data from Redis, generates signals.
5. Signals published to `paper_trade_signals:{exchange}` Redis channel.

**Files**: `strategy-engine/src/` (new: strategy_loader.rs, signal_generator.rs), `strategy-generator/src/main.rs`, `infrastructure/database/postgres/migrations/031_deployed_strategies.sql`

### B2. Paper Account Integration

**Problem**: Need to validate strategies against real market microstructure (slippage, partial fills, market impact).

**Design**:

**execution-engine changes**:
- New `ExecutionMode` enum: `Live | Paper`.
- Paper mode subscribes to `paper_trade_signals` channel.
- Routes paper orders to designated paper accounts:
  - IBKR: `ib-gateway-paper` container (paper trading account)
  - Futu: Futu paper trading mode (simulated)
  - Solana: Devnet execution (simulated)
  - Longbridge: Paper account (future)

**New DB tables** (`032_paper_trading.sql`):

**Docker**: Add `ib-gateway-paper` service to `docker-compose.yml` with paper trading credentials.

**Files**: `execution-engine/src/main.rs`, `execution-engine/src/command_listener.rs`, `docker-compose.yml`, `infrastructure/database/postgres/migrations/032_paper_trading.sql`

### B3. Strategy Promotion Criteria

**Design**:

A strategy is `promotion_eligible` when ALL conditions are met:

| Criterion | Threshold | Rationale |
|-----------|-----------|-----------|
| Paper trading days | >= 20 | Minimum statistical significance |
| Paper Sharpe ratio | > 1.0 | Risk-adjusted return threshold |
| Paper vs Backtest PnL deviation | < 20% | Signal consistency validation |
| Maximum drawdown | < 15% | Risk control |
| Fill rate | > 90% | Market liquidity confirmation |

- Promotion check runs daily as a scheduled task in execution-engine.
- Results written to `deployed_strategies.metadata` with `promotion_eligible: true/false` and individual criterion pass/fail.
- **No automatic promotion to live trading** — requires human approval via API endpoint (`POST /strategies/:id/promote`).
- Gateway exposes admin endpoint: `GET /api/v1/paper/promotion-candidates` listing eligible strategies.

### C1. Turnover-Aware Deadzone with L1 Regularization

**Problem**: Portfolio rebalance generates unnecessary trades when weight changes are trivially small.

**Design**:

**Deadzone filter** (applied after HRP + dynamic weight adjustment):
```
For each strategy i:
    delta_w[i] = |w_new[i] - w_old[i]|
    if delta_w[i] < deadzone_threshold:
        w_final[i] = w_old[i]  // suppress rebalance for this strategy
    else:
        w_final[i] = w_new[i]
Renormalize w_final to sum = 1.0
```

**L1 regularization** (in the weight optimization objective):
```
objective = max(expected_alpha_gain(w_new))
            - lambda * sum(|w_new[i] - w_old[i]|)    // L1 turnover penalty
            - sum(transaction_cost(delta_w[i]))         // explicit cost
```

**Dynamic lambda**:
- `lambda = base_lambda * (1 + vol_regime_multiplier)`
- High volatility → lower lambda (allow more rebalancing to respond to regime shift)
- Low volatility → higher lambda (suppress unnecessary trading)
- Default: `base_lambda = 0.01`

**Configuration** (`config/generator.yaml`):
```yaml
ensemble:
  deadzone:
    enabled: true
    threshold: 0.02        # 2% weight change threshold
    l1_lambda: 0.01        # L1 penalty base coefficient
    vol_adaptive: true     # scale lambda by volatility regime
```

**Files**: `strategy-generator/src/backtest/ensemble_weights.rs`, `strategy-generator/src/main.rs`, `config/generator.yaml`

### F3. Regime-Aware Rebalancing

**Problem**: Fixed 30-minute rebalance frequency is suboptimal across market regimes.

**Design**:
- Compute realized volatility: `sigma = std(returns, window=20_days) * sqrt(252)` (annualized).
- Define 3 regimes:

| Regime | Condition | Rebalance Interval |
|--------|-----------|-------------------|
| Low Vol | σ < 15% | 240 min |
| Normal | 15% ≤ σ < 30% | 60 min |
| High Vol | σ ≥ 30% | 15 min |

- Regime transition triggers immediate rebalance.
- Per-exchange regime detection (crypto vol profile differs from equities).
- Configuration: `ensemble.regime_aware: true`, `ensemble.regime_thresholds: [0.15, 0.30]`

**Files**: `strategy-generator/src/main.rs`, `strategy-generator/src/backtest/ensemble.rs`, `config/generator.yaml`

### F4. Walk-Forward Ensemble Validation

**Problem**: We validate individual strategies via walk-forward, but never the ensemble allocation itself.

**Design**:
- New function `backtest_ensemble(start_date, end_date, config)` that simulates:
  1. At each rebalance point, query strategies available at that time.
  2. Run HRP + dynamic weights using only data available at that time.
  3. Compute portfolio return for the next rebalance period.
  4. Concatenate to form ensemble equity curve.
- Runs daily at 03:00 UTC (low-load period) as a scheduled tokio task.
- Outputs: ensemble Sharpe, max drawdown, turnover, and equity curve.
- Persists to `ensemble_backtest_results` table.
- Exposed via API: `GET /:exchange/ensemble/backtest`

**Files**: `strategy-generator/src/backtest/ensemble.rs`, `strategy-generator/src/api.rs`, `infrastructure/database/postgres/migrations/033_ensemble_backtest.sql`

## 5. P6c — MCTS Symbolic Regression & Execution Optimization

### A1. MCTS Symbolic Regression Engine

**Problem**: ALPS performs undirected random search — crossover and mutation have no value awareness. In a 14-operator, 75-feature combinatorial space, this is searching blind.

**Design**:

**Module structure**: `strategy-generator/src/mcts/`
```
mcts/
    mod.rs          -- public API: MctsEngine, MctsConfig
    tree.rs         -- TreeNode, UCT selection, backpropagation
    state.rs        -- RpnState (partial formula), legal actions
    rollout.rs      -- formula completion + walk-forward evaluation
    prior.rs        -- LLM prior probability cache
    avoidance.rs    -- frequent subtree avoidance hash set
```

**Core data structures**:
```rust
struct TreeNode {
    state: RpnState,               // partial RPN formula
    action: Option<Token>,         // action that led here
    visit_count: u32,
    total_reward: f64,             // cumulative OOS PSR
    prior_probability: f32,        // P(s,a) from LLM
    children: Vec<Arc<TreeNode>>,
    parent: Option<Weak<TreeNode>>,
}

struct RpnState {
    tokens: Vec<Token>,            // built so far
    stack_depth: i32,              // current operand stack depth
    max_length: usize,             // 30 token limit
}
```

**Four-phase MCTS loop**:

1. **Selection**: Traverse tree using UCT variant:
   ```
   UCT(s,a) = Q(s,a)/N(s,a) + c_puct * P(s,a) * sqrt(N_parent) / (1 + N(s,a))
   ```
   Where `c_puct` balances exploration/exploitation (default 2.5).

2. **Expansion**: At leaf node, expand all legal actions (features, operators valid given stack state).

3. **Simulation (Rollout)**: Complete the partial formula randomly → run walk-forward backtest → obtain OOS PSR as reward.

4. **Backpropagation**: Update `visit_count` and `total_reward` on the path from leaf to root.

**LLM Policy Prior** (P(s,a)):
- At each node, query LLM: "Given partial formula `[tokens...]`, rank the next most promising tokens."
- LLM options (in priority order):
  1. Local model via llama.cpp (Qwen-2.5-7B-GGUF) for low-latency prior computation
  2. Remote model via existing LLM Oracle infrastructure (Bedrock/Anthropic)
- **Prior cache**: LRU cache keyed by `hash(RpnState)`, capacity 100K entries. Avoids redundant LLM calls for frequently visited states.
- Cache hit rate expected > 80% after initial warm-up (many searches share common prefixes).

**Risk-Seeking Policy Gradients**:
- Standard MCTS optimizes E[reward]. For alpha mining, we want to discover rare high-PSR formulas.
- Implementation: during backpropagation, only update nodes when reward is in the top 10% of all rollouts (Conditional Value at Risk upper tail).
- Effect: tree exploration is biased toward branches that occasionally produce exceptional results, not just good average results.

**Frequent Subtree Avoidance**:
- Global `HashSet<u64>` storing hashes of formula prefixes that have been explored > 100 times without PSR improvement.
- During selection, apply exploration penalty to avoided subtrees: multiply `P(s,a)` by 0.1 for actions leading into avoided territory.
- Avoidance set is reset every 10,000 MCTS iterations.

**Integration with ALPS**:
- MCTS and ALPS run in parallel as separate tokio tasks.
- Both write to the same `strategy_generations` table (MCTS results tagged with `source: "mcts"`).
- MCTS-discovered high-PSR formulas are injected into ALPS Layer 0 (same mechanism as LLM Oracle injection).
- Configuration modes:
  - `mcts.enabled: false` — ALPS only (default, backward compatible)
  - `mcts.enabled: true, mcts.replace_alps: false` — hybrid (both run)
  - `mcts.enabled: true, mcts.replace_alps: true` — MCTS only

**Configuration** (`config/generator.yaml`):
```yaml
mcts:
  enabled: false
  replace_alps: false
  iterations_per_generation: 1000
  c_puct: 2.5
  max_formula_length: 30
  risk_seeking_percentile: 0.10
  avoidance_threshold: 100
  avoidance_reset_interval: 10000
  prior:
    provider: "local"           # local | bedrock | anthropic
    local_model: "qwen2.5-7b"
    cache_capacity: 100000
```

**Estimated LoC**: 2000-3000 (new module)

**Files**: `strategy-generator/src/mcts/` (new), `strategy-generator/src/main.rs`, `strategy-generator/Cargo.toml`, `config/generator.yaml`

### A2. MCTS Search Space Constraints

**Design**:
- Maximum formula length: 30 tokens (prevents overfitting with excessively long formulas).
- Legal action mask: computed from stack state at each node.
  - Stack depth < 2 → binary operators forbidden
  - Stack depth = 1 and formula complete → only termination allowed
  - Stack depth > 8 → only operators allowed (force stack reduction)
- Token types: `Feature(0..74)`, `Constant(f64)`, `UnaryOp`, `BinaryOp`, `TsOp(window)`
- Termination condition: `stack_depth == 1` and no unclosed TS operations.
- Constants: drawn from `{0.5, 1.0, 2.0, 5.0, 10.0, 20.0}` (discrete set, not continuous).

### D1. Execution Layer Performance Optimization

**Problem**: Live order routing needs minimized tail latency for time-sensitive markets.

**Design** (three levels, implemented incrementally):

**Level 1 — Immediate wins** (no architecture change):
- Profile execution-engine signal processing path with `cargo flamegraph`.
- Pre-allocate order struct pool (`Vec<Order>` with capacity 1000) at startup. Reuse via index.
- Dedicated Tokio runtime for Redis signal subscription (isolate from order execution).
- Configuration: `EXEC_ENGINE_REDIS_THREADS=2`

**Level 2 — Lock-free internal communication**:
- Replace internal Redis pub/sub (within same process) with `crossbeam::channel::bounded` SPSC channel for signal → execution handoff.
- Retain Redis for cross-service communication (strategy-engine → execution-engine).
- Expected latency reduction: ~1-5ms per signal (eliminates Redis round-trip for internal routing).

**Level 3 — Kernel-level optimization** (Linux production only):
- `core_affinity` crate: pin I/O thread to dedicated CPU core.
- `io_uring` via `tokio-uring`: replace epoll for network I/O on Linux 5.6+.
- Zero heap allocation during trading hours: all buffers pre-allocated at startup.
- **Not applicable to macOS dev environment** — gated by `#[cfg(target_os = "linux")]`.

**Files**: `execution-engine/src/main.rs`, `execution-engine/src/command_listener.rs`, `execution-engine/Cargo.toml`

### D2. Multi-Exchange Portfolio Allocation

**Problem**: Current HRP ensemble runs per-exchange. No cross-exchange diversification.

**Design**:
- Extend ensemble rebalance to support `exchange = "ALL"` mode.
- Load candidates from all exchanges into a single return matrix.
- Time axis normalization for mixed trading hours:

| Venue Type | Trading Hours | Normalization |
|-----------|---------------|---------------|
| Crypto (Binance, OKX, Bybit, Solana) | 24/7 | Use all bars |
| US Stocks (IBKR) | 09:30-16:00 ET, M-F | Align to business-day bars only |
| HK Stocks (Futu, Longbridge) | 09:30-16:00 HKT, M-F | Align to business-day bars only |
| A-Shares (future) | 09:30-15:00 CST, M-F | Align to business-day bars only |

- For cross-exchange correlation: resample all returns to daily frequency (close-to-close), which naturally handles different trading hours.
- HRP runs on the daily return matrix across all venues.
- Weight allocation respects per-exchange capital limits (configurable):
  ```yaml
  ensemble:
    multi_exchange:
      enabled: false
      capital_limits:
        IBKR: 0.40       # max 40% of portfolio
        Binance: 0.30
        Futu: 0.20
        Solana: 0.10
  ```
- Per-exchange ensembles continue to run alongside the global ensemble.
- Extensible: adding a new exchange (Longbridge, A-share brokers) only requires adding a new entry to `capital_limits`.

**Files**: `strategy-generator/src/backtest/ensemble.rs`, `strategy-generator/src/main.rs`, `config/generator.yaml`

## 10. Risks & Mitigations

| Risk | Severity | Phase | Mitigation |
|------|----------|-------|------------|
| MCTS search space explosion | High | P6c | Max formula length=30, legal action mask, frequent subtree avoidance |
| LLM prior latency blocks MCTS | Medium | P6c | Prior cache (100K entries), async prefetch, fallback to uniform prior |
| Paper trading data leak to backtest | High | P6b | Strict table isolation (paper_* tables), separate DB schemas if needed |
| Atomic blocks fossilize bad strategies | Medium | P6a | Block decay after 20 gens without improvement |
| rkyv schema incompatibility | Low | P6a | Version prefix byte, dual-mode reader, only deploy after profiling |
| IBKR paper account rate limits | Low | P6b | Throttle order submission, batch orders |
| Multi-exchange time alignment errors | Medium | P6c | Use daily frequency for cross-exchange correlation, unit test edge cases |
